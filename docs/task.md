# 每条订单数据格式

`raw_orders/<订单号>-2.json` 是一个订单级排产输入文件。每个文件通常包含下面 5 个顶层字段：

## 当前订单信息

`当前订单信息` 是订单需求列表，每一行表示一个产品需求：

- `产品名称`：产品 ID。
- `需求量`：该产品需要交付的数量。
- `期限`：相对排产天数，表示最晚需要在第几天完成。

## 产品工序

`产品工序` 是产品到工艺路线的映射，key 形如 `<产品ID>工艺信息`，value 是该产品的工序列表。每道工序包含：

- `序号`：工序顺序，从 1 开始。
- `工序`：工序名称。
- `所用设备`：该工序需要占用的设备列表；`无` 表示手工作业，不占用设备。如果列表里有多个非 `无` 设备，表示这些设备需要被同时占用，不是从中任选一个。
- `耗时`：单件产品完成该工序需要的时间，单位按分钟使用。
- `可选操作人员`：可以执行该工序的工人候选集合。

排产时，每个 task 会为该工序选择 1 名工人；如果该工序需要设备，需要同时占用 `所用设备` 中列出的全部非 `无` 设备各一台。后续工序会消耗上一道工序产出的中间产物，不能凭空跳过前序工序。

可以把工序执行理解为单位产物在工艺链上的流转：执行某产品第 1 道工序时，由指定工人操作该工序所需设备并持续 `耗时` 分钟，产出 1 份第 1 道工序后的中间产物；执行第 2 道及之后的工序时，会消耗 1 份上一道工序产出的中间产物，并产出 1 份当前工序后的中间产物。若一次排产 task 的 `quantity=q`，则消耗和产出都按 `q` 份计算，总占用工时为 `耗时 * q`。

## 相关产品库存

`相关产品库存` 是产品 ID 到初始库存数量的映射。当前代码主要把它当作初始成品库存，用来抵扣订单需求；排产过程中还会维护各产品各工序的中间库存。

## 可使用设备信息

`可使用设备信息` 是设备主数据列表：

- `设备名称`：设备名称，必须能覆盖工序里的非 `无` 设备引用。
- `数量`：该设备可用台数。

订单数据没有给出设备每日工作时长上限。业务口径下，设备需要存在并被正确占用，但不按每天 480 分钟限制设备产能；480 分钟上限只用于工人每日工作时间。

## 每日可使用人员列表

`每日可使用人员列表` 是人员排班表，结构是 `工人姓名 -> 可上班天数列表`。只有出现在对应天数列表中的工人，才能在该天被排产。

每个工人在每个可用日最多工作 480 分钟。排给同一工人的所有 task 会累计占用工时，超过 480 分钟会校验失败。

## 工序约束摘要

- 每道工序的 `耗时` 是单件分钟数，task 总工时为 `耗时 * quantity`。
- 每个 task 必须指定 1 名满足 `可选操作人员` 的工人。
- 设备工序必须占用 `所用设备` 中列出的全部非 `无` 设备；手工作业使用 `无`。
- 第 2 道及之后的工序数量，不能超过历史中间库存加当天前一道工序产出的数量。
- 工人当天累计占用时间不能超过 480 分钟。

注意：这里描述的是订单数据的业务语义。若排产工具或校验代码仍使用单个 `machine` 字段，或仍给设备套用 `数量 * 480` 的每日产能限制，需要同步改造，否则会低估设备占用关系或额外引入设备产能约束。
